## Entity Relationship Overview

```
auth.users (Supabase built-in)
    │  ON INSERT → handle_new_user trigger
    ▼
users (public)
    │  user_id (nullable — NULL for guests)
    ▼
orders ──────────── order_items
    │                   │
    │                   ├── product_id → products (ON DELETE SET NULL)
    │                   └── variant_id → product_variants (ON DELETE SET NULL)
    │
    ├── zone_id → delivery_zones
    │
    └── point_transactions (user_id → users)

categories ──── products ──── product_variants
offers                          (independent)
settings                        (key-value store)
```

**Key design facts:**
- `orders.user_id = NULL` for guest orders — no `public.users` record is created for guests.
- `order_items` uses `ON DELETE SET NULL` on both `product_id` and `variant_id` so that product deletion does not destroy historical order records.
- `guest_token` on `orders` is a UUID generated in `sessionStorage` on first visit — used exclusively for guest-to-registered merge (never phone number alone).
- All `points_balance` mutations go through DB triggers or RPCs — never direct `UPDATE users SET points_balance`.

### `products`

```sql
CREATE TABLE products (
  id            uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id   uuid        REFERENCES categories(id) ON DELETE CASCADE,
  name          text        NOT NULL,
```

### `users`

```sql
CREATE TABLE users (

  auth_id          uuid    UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,
  phone_number     text    UNIQUE NOT NULL,
  name             text,

  points_balance   integer DEFAULT 0,
  created_at       timestamp DEFAULT now()
);
```

**Auth strategy**: Supabase Email Auth using synthetic address `{phone}@khaldoun.local`. Phone number stored in `raw_user_meta_data.phone_number` at signup. The `handle_new_user` trigger auto-populates this table on every `auth.users` insert.

**Points constraint**: `points_balance` MUST only be modified through `handle_order_status_change` trigger or `create_order_with_points` / `admin_adjust_points` RPCs. Direct `UPDATE users SET points_balance = ...` from application code is a constitution violation (Principle I).

### `orders`

```sql
CREATE TABLE orders (
  id                uuid    PRIMARY KEY DEFAULT gen_random_uuid(),
  order_code        text    UNIQUE NOT NULL,          -- generated by trigger: "KH-1042"
  user_id           uuid    REFERENCES users(id),     -- NULL for guest orders
  customer_name     text    NOT NULL,                 -- snapshot, independent of auth account
  customer_phone    text    NOT NULL,
  delivery_address  text,                             -- NULL for pickup orders
  order_type        text    NOT NULL,                 -- 'delivery' | 'pickup'
  zone_id           uuid    REFERENCES delivery_zones(id),  -- required for delivery
  notes             text,
  status            text    DEFAULT 'pending',        -- pending | confirmed | preparing | out_for_delivery | delivered | cancelled
  subtotal          integer NOT NULL,
  delivery_fee      integer DEFAULT 0,
  discount_amount   integer DEFAULT 0,                -- points discount in EGP
  total_price       integer NOT NULL,
  points_used       integer DEFAULT 0,                -- points deducted at order time
  points_earned     integer DEFAULT 0,                -- points to be awarded on confirmed
  points_status     text    DEFAULT 'pending',        -- 'pending' | 'confirmed' | 'cancelled'
  whatsapp_opened   boolean DEFAULT false,            -- wa.me opens app only, cannot confirm send
  guest_token       text,                             -- UUID from sessionStorage for guest merge
  created_at        timestamp DEFAULT now(),

  CONSTRAINT chk_zone_required CHECK (order_type = 'pickup' OR zone_id IS NOT NULL),
  CONSTRAINT chk_status CHECK (status IN ('pending', 'confirmed', 'preparing', 'out_for_delivery', 'delivered', 'cancelled'))
);
```

**Note on `whatsapp_opened`**: Named `whatsapp_opened`, not `whatsapp_sent` — `wa.me` opens the WhatsApp app with a pre-filled message but cannot confirm whether the user pressed Send.

**Note on `is_guest_order`**: Column intentionally omitted. Use `(user_id IS NULL)` predicate in all queries to identify guest orders.

**`chk_zone_required`**: Delivery orders must have a `zone_id`; pickup orders may have `zone_id = NULL`.

### `order_items`

```sql
CREATE TABLE order_items (
  id            uuid      PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id      uuid      REFERENCES orders(id) ON DELETE CASCADE,
  product_id    uuid      REFERENCES products(id) ON DELETE SET NULL,   -- nullable after product deletion
  variant_id    uuid      REFERENCES product_variants(id) ON DELETE SET NULL,  -- nullable
  product_name  text      NOT NULL,   -- SNAPSHOT: captured at order time — immutable
  variant_name  text,                 -- SNAPSHOT: captured at order time — immutable
  unit_price    integer   NOT NULL,   -- SNAPSHOT: captured at order time — immutable
  quantity      integer   NOT NULL DEFAULT 1,
  subtotal      integer   NOT NULL,   -- unit_price × quantity

  CONSTRAINT chk_order_item_subtotal CHECK (subtotal = unit_price * quantity)
);
```

**Snapshot immutability (Constitution Principle IV)**: `product_name`, `variant_name`, and `unit_price` are written once at order insertion and MUST NOT be updated afterward. If a product is deleted, `product_id` becomes NULL but the snapshot fields remain intact and fully readable for historical order display.

### `point_transactions`

```sql
CREATE TABLE point_transactions (
  id                uuid      PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           uuid      REFERENCES users(id) ON DELETE CASCADE,
  order_id          uuid      REFERENCES orders(id),  -- nullable for manual adjustments
  transaction_type  text      NOT NULL,
    -- 'earned'        → points added when order confirmed
    -- 'redeemed'      → points deducted at order placement
    -- 'manual_add'    → admin manually added points
    -- 'manual_deduct' → admin manually deducted points
    -- 'refunded'      → automatic refund of points_used on order cancellation
    -- 'reversed'      → automatic reversal of points_earned on confirmed→cancelled
  points            integer   NOT NULL CHECK (points > 0),  -- always positive; direction = transaction_type
  note              text,
  created_at        timestamp DEFAULT now()
);
```

**Direction convention**: `points` is always positive. The `transaction_type` determines whether it adds to or subtracts from `points_balance`. This avoids redundancy and clarifies intent.

## Triggers

### Trigger 1: `handle_new_user`

Auto-populates `public.users` whenever a new Supabase Auth user is created.

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.users (auth_id, phone_number, name)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'phone_number', ''),
    COALESCE(NEW.raw_user_meta_data->>'name', '')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

**Why SECURITY DEFINER**: Runs with table-owner privileges to insert into `public.users` regardless of the calling user's RLS context.

### Trigger 3: `handle_order_status_change`

Handles all loyalty points side effects when an order's status changes. App code only sets the new status — this trigger does everything else.

```sql
CREATE OR REPLACE FUNCTION handle_order_status_change()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  -- On confirmed: award earned points to user balance
  IF OLD.status != 'confirmed' AND NEW.status = 'confirmed'
     AND NEW.points_earned > 0 AND NEW.user_id IS NOT NULL THEN
    UPDATE users
      SET points_balance = points_balance + NEW.points_earned
      WHERE id = NEW.user_id;
    INSERT INTO point_transactions (user_id, order_id, transaction_type, points)
      VALUES (NEW.user_id, NEW.id, 'earned', NEW.points_earned);
    NEW.points_status := 'confirmed';
  END IF;

  -- On cancellation of a previously confirmed order: reverse earned points
  IF OLD.status = 'confirmed' AND NEW.status = 'cancelled'
     AND OLD.points_earned > 0 AND OLD.user_id IS NOT NULL THEN
    UPDATE users
      SET points_balance = GREATEST(0, points_balance - OLD.points_earned)
      WHERE id = OLD.user_id;
    INSERT INTO point_transactions (user_id, order_id, transaction_type, points, note)
      VALUES (OLD.user_id, OLD.id, 'reversed', OLD.points_earned,
              'تراجع — إلغاء طلب ' || OLD.order_code);
    NEW.points_status := 'cancelled';
  END IF;

  -- On any cancellation: refund points_used (the discount the customer took)
  IF NEW.status = 'cancelled' AND OLD.status != 'cancelled'
     AND OLD.points_used > 0 AND OLD.user_id IS NOT NULL THEN
    UPDATE users
      SET points_balance = points_balance + OLD.points_used
      WHERE id = OLD.user_id;
    INSERT INTO point_transactions (user_id, order_id, transaction_type, points, note)
      VALUES (OLD.user_id, OLD.id, 'refunded', OLD.points_used,
              'استرداد خصم — إلغاء طلب ' || OLD.order_code);
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER on_order_status_change
  BEFORE UPDATE OF status ON orders
  FOR EACH ROW EXECUTE FUNCTION handle_order_status_change();
```

**Covers all three cases**:
1. `pending → confirmed`: Award `points_earned` to user balance.
2. `confirmed → cancelled`: Reverse `points_earned` from user balance.
3. Any `→ cancelled`: Refund `points_used` (the discount) back to user balance.

## RPC: `create_order_with_points`

Atomic order creation for registered users who are redeeming points. Uses `FOR UPDATE` row lock to prevent race conditions.

```sql
CREATE OR REPLACE FUNCTION create_order_with_points(
  p_user_id       uuid,
  p_points_to_use integer,
  p_subtotal      integer,
  p_delivery_fee  integer,
  p_discount      integer,
  p_total         integer,
  p_order_data    jsonb,  -- { customer_name, customer_phone, delivery_address, order_type, zone_id, notes }
  p_items         jsonb   -- [{ product_id, variant_id, product_name, variant_name, unit_price, quantity }]
) RETURNS uuid LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_balance        integer;
  v_order_id       uuid;
  v_points_per_100 integer;
  v_points_earned  integer;
  v_item           jsonb;
BEGIN
  -- Guard: prevent negative points input (could add instead of deduct)
  IF p_points_to_use < 0 THEN
    RAISE EXCEPTION 'invalid_points: لا يجوز نقاط سالبة';
  END IF;

  -- Acquire row lock — blocks concurrent transactions on this user
  SELECT points_balance INTO v_balance
  FROM users WHERE id = p_user_id FOR UPDATE;

  IF v_balance < p_points_to_use THEN
    RAISE EXCEPTION 'insufficient_points';
  END IF;

  -- Calculate earned points from settings (computed on subtotal only)
  SELECT COALESCE(CAST(value AS integer), 1)
  INTO v_points_per_100
  FROM settings WHERE key = 'points_per_100_egp';

  v_points_earned := FLOOR(p_subtotal::numeric / 100) * COALESCE(v_points_per_100, 1);

  -- Insert order
  INSERT INTO orders (
    user_id, customer_name, customer_phone, delivery_address,
    order_type, zone_id, notes,
    subtotal, delivery_fee, discount_amount, total_price,
    points_used, points_earned
  ) VALUES (
    p_user_id,
    p_order_data->>'customer_name',
    p_order_data->>'customer_phone',
    p_order_data->>'delivery_address',
    p_order_data->>'order_type',
    NULLIF(p_order_data->>'zone_id', '')::uuid,
    p_order_data->>'notes',
    p_subtotal, p_delivery_fee, p_discount, p_total,
    p_points_to_use, v_points_earned
  ) RETURNING id INTO v_order_id;

  -- Insert order_items with snapshots
  FOR v_item IN SELECT * FROM jsonb_array_elements(p_items) LOOP
    INSERT INTO order_items (
      order_id, product_id, variant_id,
      product_name, variant_name,
      unit_price, quantity, subtotal
    ) VALUES (
      v_order_id,
      (v_item->>'product_id')::uuid,
      NULLIF(v_item->>'variant_id', '')::uuid,
      v_item->>'product_name',
      NULLIF(v_item->>'variant_name', ''),
      (v_item->>'unit_price')::integer,
      (v_item->>'quantity')::integer,
      (v_item->>'unit_price')::integer * (v_item->>'quantity')::integer
    );
  END LOOP;

  -- Deduct points atomically
  IF p_points_to_use > 0 THEN
    UPDATE users SET points_balance = points_balance - p_points_to_use
    WHERE id = p_user_id;
    INSERT INTO point_transactions (user_id, order_id, transaction_type, points)
    VALUES (p_user_id, v_order_id, 'redeemed', p_points_to_use);
  END IF;

  -- Record earned points only if > 0 (subtotal < 100 EGP yields 0 points)
  IF v_points_earned > 0 THEN
    INSERT INTO point_transactions (user_id, order_id, transaction_type, points)
    VALUES (p_user_id, v_order_id, 'earned', v_points_earned);
  END IF;

  RETURN v_order_id;
END;
$$;
```

## Key Design Decisions

### 1. Snapshot Immutability (Constitution Principle IV)

`order_items.product_name`, `.variant_name`, and `.unit_price` are written once at order creation and are never updated. This ensures every historical order can be displayed exactly as it was placed, regardless of subsequent admin changes to the product catalogue.

### 2. `ON DELETE SET NULL` for Product References

`order_items.product_id` and `order_items.variant_id` use `ON DELETE SET NULL` rather than `ON DELETE CASCADE` or `ON DELETE RESTRICT`. This allows admins to permanently delete discontinued products without destroying historical order records. The snapshot fields remain intact.

### 3. `guest_token` for Secure Guest Merge

A UUID is generated in `sessionStorage` on the customer's first page visit and stored with every guest order as `guest_token`. When a guest subsequently creates an account, the Server Action merges their orders using:
```sql
UPDATE orders SET user_id = $newUserId
WHERE guest_token = $token AND user_id IS NULL
```
Phone number match alone MUST NOT trigger a merge — this prevents account takeover via phone number spoofing (Constitution Principle I).

### 4. `user_id IS NULL` for Guest Detection

The column `is_guest_order` is intentionally absent from `orders`. The predicate `user_id IS NULL` is used directly in all queries that need to distinguish guest orders from registered-user orders.

### 5. Points Always Positive in `point_transactions`

The `points` column in `point_transactions` is always positive (`CHECK (points > 0)`). The `transaction_type` field (`earned`, `redeemed`, `manual_add`, `manual_deduct`, `refunded`, `reversed`) determines the direction of the balance change. This avoids negative values in the ledger and makes audit queries simpler.

## RPC: `admin_adjust_points`

Manual points adjustment by admin. Updates `points_balance` and inserts a `point_transactions` record atomically.

```sql
CREATE OR REPLACE FUNCTION admin_adjust_points(
  p_user_id   uuid,
  p_amount    integer,
  p_type      text,    -- 'manual_add' | 'manual_deduct'
  p_note      text DEFAULT NULL
) RETURNS void LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_balance integer;
BEGIN
  -- Validate inputs
  IF p_amount <= 0 THEN
    RAISE EXCEPTION 'invalid_amount: \u064a\u062c\u0628 \u0623\u0646 \u064a\u0643\u0648\u0646 \u0627\u0644\u0645\u0628\u0644\u063a \u0623\u0643\u0628\u0631 \u0645\u0646 \u0635\u0641\u0631';
  END IF;
  IF p_type NOT IN ('manual_add', 'manual_deduct') THEN
    RAISE EXCEPTION 'invalid_type: \u064a\u062c\u0628 \u0623\u0646 \u064a\u0643\u0648\u0646 \u0627\u0644\u0646\u0648\u0639 manual_add \u0623\u0648 manual_deduct';
  END IF;

  -- Acquire row lock to prevent concurrent adjustments
  SELECT points_balance INTO v_balance
  FROM users WHERE id = p_user_id FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'user_not_found';
  END IF;

  IF p_type = 'manual_deduct' AND v_balance < p_amount THEN
    RAISE EXCEPTION 'insufficient_points';
  END IF;

  -- Update balance
  IF p_type = 'manual_add' THEN
    UPDATE users SET points_balance = points_balance + p_amount WHERE id = p_user_id;
  ELSE
    UPDATE users SET points_balance = GREATEST(0, points_balance - p_amount) WHERE id = p_user_id;
  END IF;

  -- Record transaction
  INSERT INTO point_transactions (user_id, order_id, transaction_type, points, note)
  VALUES (p_user_id, NULL, p_type, p_amount, p_note);
END;
$$;
```

**Security**: `SECURITY DEFINER` allows the function to bypass RLS and directly update `users.points_balance` — this is the ONLY permitted application-layer path for manual points adjustments (Constitution Principle I).
