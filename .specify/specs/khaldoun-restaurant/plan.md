## Summary

Build a complete Arabic-RTL online ordering website for Khaldoun Restaurant (Egypt). Customers browse a dynamic menu, add items with variants to a persisted cart, and submit orders via a formatted WhatsApp deep-link. Registered users earn and redeem loyalty points. An admin panel provides full control over orders, menu, settings, and customers — all without redeployment. Stack: Next.js 15 App Router + Supabase + Tailwind CSS v4 + Zustand v5, hosted on Vercel.

---

## Technical Context

**Language/Version**: TypeScript 5.x / Next.js 15.x (App Router + Turbopack)
**Primary Dependencies**: Next.js 15, Tailwind CSS v4, @supabase/supabase-js, @supabase/ssr, Zustand v5, React Hook Form, Zod, Lucide React
**Storage**: Supabase PostgreSQL (10 tables) + Supabase Storage bucket `menu-images`
**Testing**: Manual E2E — no automated tests requested per spec
**Target Platform**: Web — mobile-first, Vercel hosting
**Project Type**: web-service / e-commerce ordering system
**Performance Goals**: <2s LCP on 3G mobile (Egypt); images served via Supabase Image Transform `?width=400&quality=75`
**Constraints**: Arabic RTL only (`dir="rtl"` + `lang="ar"`); cash payment only; WhatsApp ordering via `wa.me` deep-link (no API key); WhatsApp URL must stay ≤2048 chars after `encodeURIComponent`
**Scale/Scope**: Single restaurant; ~100–500 orders/day; ~1,000 registered users

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

| المبدأ | الحالة | الدليل |
|--------|--------|--------|
| **I. Security-First** | ✅ PASS | `SUPABASE_SERVICE_ROLE_KEY` used only in Server Actions and Route Handlers — never in `NEXT_PUBLIC_`. All `points_balance` mutations route through `handle_order_status_change` trigger or `create_order_with_points` RPC exclusively. Admin role granted only via `app_metadata: { role: "admin" }` checked by RLS policy `auth.jwt() -> 'app_metadata' ->> 'role'`. Guest-order merge uses `guest_token` (session UUID) only — phone match alone is explicitly prohibited. |
| **II. Mobile-First Arabic UI** | ✅ PASS | Root `<html dir="rtl" lang="ar">`. Primary font: Cairo or Tajawal (Google Fonts, Arabic-optimised). All interactive targets `min-h-[44px]`. Navigation uses Bottom Sheet pattern on mobile. Tailwind v4 built-in RTL — no third-party plugin. |
| **III. Data Integrity via DB Layer** | ✅ PASS | Points accrual on `confirmed` status and reversal on `cancelled` handled by `handle_order_status_change` trigger. Atomic order creation + points deduction via `create_order_with_points` RPC with `FOR UPDATE` row lock. `order_code` generated by `generate_order_code` trigger (prefix `KH-`). App code only calls `UPDATE orders SET status = $new` — trigger handles all side effects. |
| **IV. Snapshot Immutability** | ✅ PASS | Every `order_items` row stores `product_name`, `variant_name`, `unit_price` as snapshots at insertion time. References use `ON DELETE SET NULL` — product deletion nullifies FK but preserves snapshot fields. No retroactive price correction to past `order_items`. |
| **V. Dynamic Settings without Redeployment** | ✅ PASS | All operator-configurable values (WhatsApp numbers, social links, points rules, delivery fees, banners) stored in `settings` table. Caching via `unstable_cache` from `'next/cache'` with `tags: ['settings']` + `revalidateTag('settings')` on admin save for immediate cross-request invalidation. No hardcoded values in source files or env vars. |

**All 5 gates: PASS — no violations to justify.**
